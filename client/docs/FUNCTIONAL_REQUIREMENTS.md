# Функциональные требования

## 1. Общие сведения

Документ описывает функциональные требования к MVP-версии мобильной игры **Call of the Forest** и связанному с ней backend-сервису.

Проект представляет собой 2D survival-игру для Android. Игрок управляет персонажем в лесном мире, собирает ресурсы, сражается с мобами, следит за здоровьем и голодом, переживает смену дня и ночи, а после завершения партии может отправить результат в лидерборд.

## 2. Роли пользователей

| Роль | Описание |
| :--- | :--- |
| Гость | Пользователь без авторизации. Может открыть приложение, перейти на экран авторизации и просмотреть общий лидерборд. |
| Авторизованный игрок | Пользователь, прошедший регистрацию или вход. Может запускать игру, отправлять результаты партии, просматривать лидерборд и личную статистику. |

## 3. Основные сценарии

### 3.1. Регистрация пользователя

1. Пользователь открывает экран входа.
2. Пользователь вводит имя пользователя и пароль.
3. Пользователь выбирает режим регистрации.
4. Клиент отправляет данные на backend.
5. Backend создает пользователя, возвращает access token, refresh token и имя пользователя.
6. Клиент сохраняет данные сессии локально.
7. Пользователь возвращается в главное меню как авторизованный игрок.

### 3.2. Вход пользователя

1. Пользователь открывает экран входа.
2. Пользователь вводит имя пользователя и пароль.
3. Пользователь выбирает вход.
4. Клиент отправляет данные на backend.
5. Backend проверяет учетные данные и возвращает токены.
6. Клиент сохраняет данные сессии локально.
7. Пользователь получает доступ к запуску игры.

### 3.3. Запуск партии

1. Авторизованный игрок открывает главное меню.
2. Игрок нажимает кнопку `Play`.
3. Клиент создает новое игровое состояние.
4. Игровой цикл запускается и начинает обновлять состояние мира.
5. Игрок видит карту, персонажа, мобов, объекты и HUD.

### 3.4. Игровой процесс

1. Игрок перемещает персонажа с помощью виртуального джойстика.
2. Камера следует за персонажем.
3. Игрок исследует карту и взаимодействует с объектами.
4. Игрок собирает ресурсы с деревьев, камней и кустов.
5. Игрок может атаковать мобов в ближнем бою.
6. Мобы реагируют на игрока в зависимости от типа поведения.
7. Показатели здоровья и голода изменяются во времени.
8. При снижении здоровья до нуля партия завершается.

### 3.5. Завершение партии и отправка результата

1. При смерти персонажа игра переходит в состояние `Game Over`.
2. Клиент показывает итоговую статистику партии.
3. Если пользователь авторизован, клиент отправляет результат на backend.
4. Backend рассчитывает итоговый score и сохраняет результат.
5. Пользователь может вернуться в меню или повторить отправку результата при ошибке сети.

### 3.6. Просмотр лидерборда

1. Пользователь открывает экран лидерборда.
2. Клиент запрашивает топ результатов с backend.
3. Backend возвращает список лучших результатов.
4. Клиент отображает место, имя игрока, score и основные показатели партии.
5. Если пользователь авторизован, клиент дополнительно запрашивает личную статистику.

## 4. Требования к клиентскому приложению

### FR-C-001. Главное меню

Приложение должно отображать главное меню с названием игры и основными действиями.

Для гостя должны быть доступны:

- переход на экран входа;
- просмотр лидерборда;
- выход из приложения.

Для авторизованного игрока должны быть доступны:

- запуск игры;
- просмотр лидерборда;
- выход из аккаунта;
- выход из приложения.

### FR-C-002. Экран авторизации

Приложение должно предоставлять экран для регистрации и входа пользователя.

Экран должен позволять:

- ввести имя пользователя;
- ввести пароль;
- выполнить регистрацию;
- выполнить вход;
- вернуться в главное меню.

При успешной авторизации приложение должно сохранить access token, refresh token и имя пользователя.

### FR-C-003. Хранение сессии

Приложение должно локально хранить данные пользовательской сессии в `SharedPreferences`.

Приложение должно использовать сохраненный access token при обращении к защищенным backend endpoint'ам.

Приложение должно позволять пользователю выйти из аккаунта. При выходе данные сессии должны удаляться.

### FR-C-004. Запуск новой игры

При запуске партии приложение должно создать новое состояние игры, включающее:

- игрока;
- камеру;
- игровые объекты карты;
- мобов;
- время суток;
- статистику партии;
- состояние здоровья, голода и инвентаря.

### FR-C-005. Игровой цикл

Приложение должно обновлять игровое состояние в цикле во время активной партии.

Игровой цикл должен выполнять:

- обработку движения игрока;
- обработку атаки игрока;
- обработку сбора ресурсов;
- обработку употребления ягод;
- обновление поведения мобов;
- обработку столкновений;
- обновление голода, здоровья и времени суток;
- проверку условия окончания партии;
- обновление визуальных эффектов.

### FR-C-006. Управление игроком

Приложение должно позволять игроку перемещать персонажа с помощью виртуального джойстика.

Персонаж должен:

- двигаться в направлении ввода;
- иметь ограниченную скорость;
- менять направление взгляда при движении;
- останавливаться при отсутствии ввода.

### FR-C-007. Камера

Камера должна следовать за персонажем игрока и смещать отображение мира относительно текущей позиции игрока.

### FR-C-008. Отрисовка игрового мира

Приложение должно отрисовывать игровой мир через Compose Canvas.

На игровом экране должны отображаться:

- фон карты;
- персонаж игрока;
- деревья;
- камни;
- кусты;
- мобы;
- эффекты атаки;
- HUD с показателями игрока и партии.

### FR-C-009. Генерация карты

При создании партии приложение должно сформировать набор объектов карты и мобов.

На карте должны присутствовать:

- деревья;
- камни;
- ягодные кусты;
- хищные и мирные мобы.

### FR-C-010. Коллизии

Приложение должно обрабатывать базовые столкновения между сущностями.

Коллизии должны препятствовать некорректному наложению игрока, объектов карты и мобов.

### FR-C-011. Сбор ресурсов

Приложение должно позволять игроку собирать ресурсы с объектов карты при нахождении рядом с объектом.

Тип ресурса должен зависеть от типа объекта:

| Объект | Ресурс |
| :--- | :--- |
| Дерево | Wood |
| Камень | Stone |
| Куст | Berry |

После успешного сбора количество ресурса в инвентаре должно увеличиваться.

### FR-C-012. Инвентарь

Приложение должно хранить ресурсы игрока в инвентаре.

Инвентарь должен поддерживать следующие типы ресурсов:

- wood;
- stone;
- berry;
- meat.

HUD должен показывать количество ресурсов игрока.

### FR-C-013. Употребление ягод

Приложение должно позволять игроку употребить ягоду из инвентаря.

При употреблении ягоды:

- количество ягод должно уменьшиться на 1;
- показатель голода должен восстановиться;
- значение голода не должно превышать максимальное значение.

Если ягод нет, действие не должно изменять состояние игры.

### FR-C-014. Здоровье и голод

Приложение должно хранить показатели здоровья и голода игрока.

Голод должен постепенно уменьшаться во время партии.

При достижении нулевого голода игрок должен получать периодический урон.

При снижении здоровья до нуля партия должна завершиться.

### FR-C-015. Смена дня и ночи

Приложение должно поддерживать циклическую смену времени суток.

HUD должен отображать текущее время или фазу суток.

Ночью игрок должен получать дополнительное воздействие холода, влияющее на здоровье.

### FR-C-016. Мобы

Приложение должно поддерживать несколько типов мобов:

- wolf;
- bear;
- fox;
- deer;
- rabbit;
- bird.

Каждый тип моба должен иметь собственные параметры скорости, здоровья, радиуса обнаружения, радиуса атаки и урона.

### FR-C-017. Поведение мобов

Мобы должны обновлять поведение во время игрового цикла.

Хищные мобы должны уметь:

- находиться в состоянии ожидания;
- патрулировать;
- преследовать игрока;
- атаковать игрока при сближении.

Мирные мобы должны уметь:

- находиться в состоянии ожидания;
- патрулировать;
- убегать от игрока при приближении.

### FR-C-018. Боевая система

Приложение должно позволять игроку атаковать мобов в ближнем бою.

Атака должна:

- иметь ограниченный радиус действия;
- учитывать направление игрока;
- наносить урон мобам в зоне атаки;
- иметь cooldown;
- создавать визуальный эффект атаки.

При смерти моба статистика убитых мобов должна увеличиваться.

### FR-C-019. Добыча мяса с мобов

При убийстве моба приложение может добавить мясо в инвентарь игрока.

Вероятность выпадения мяса должна зависеть от типа моба.

### FR-C-020. HUD

Игровой экран должен отображать основные показатели партии:

- здоровье игрока;
- голод игрока;
- время суток;
- количество ресурсов;
- статистику партии;
- доступные действия игрока.

### FR-C-021. Пауза

Приложение должно позволять поставить игру на паузу и продолжить партию.

Во время паузы игровой цикл не должен изменять состояние партии.

### FR-C-022. Завершение партии

При смерти игрока приложение должно:

- остановить активное изменение игрового состояния;
- показать экран или overlay `Game Over`;
- показать статистику партии;
- инициировать отправку результата, если игрок авторизован.

### FR-C-023. Статистика партии

Приложение должно вести статистику партии:

- время выживания в секундах;
- количество убитых мобов;
- количество собранных ресурсов;
- количество пережитых дней.

Эта статистика должна использоваться для отправки результата на backend.

### FR-C-024. Отправка результата партии

После завершения партии приложение должно отправлять результат на backend для авторизованного пользователя.

В запрос должны входить:

- survivedSeconds;
- mobsKilled;
- resourcesGathered;
- daysSurvived.

Если пользователь не авторизован, отправка результата должна пропускаться.

Если при отправке произошла ошибка, приложение должно показать ошибку и позволить повторить отправку.

### FR-C-025. Экран лидерборда

Приложение должно отображать экран лидерборда.

Экран должен показывать:

- общий топ результатов;
- имя игрока;
- score;
- время выживания;
- количество убитых мобов;
- количество собранных ресурсов;
- количество пережитых дней.

Для авторизованного пользователя экран должен дополнительно показывать личную статистику, если она доступна.

## 5. Требования к backend

### FR-S-001. Регистрация

Backend должен предоставлять endpoint для регистрации пользователя.

Endpoint должен принимать имя пользователя и пароль.

При успешной регистрации backend должен вернуть:

- access token;
- refresh token;
- username.

### FR-S-002. Вход

Backend должен предоставлять endpoint для входа пользователя.

Endpoint должен проверять имя пользователя и пароль.

При успешном входе backend должен вернуть access token, refresh token и username.

### FR-S-003. Обновление access token

Backend должен предоставлять endpoint для обновления access token по refresh token.

При валидном refresh token backend должен вернуть новую пару токенов.

### FR-S-004. Получение текущего пользователя

Backend должен предоставлять защищенный endpoint для получения данных текущего пользователя.

Endpoint должен возвращать:

- id пользователя;
- username;
- role.

### FR-S-005. JWT-авторизация

Backend должен проверять JWT для защищенных endpoint'ов.

При отсутствии или некорректности токена backend должен отклонять запрос.

### FR-S-006. Хеширование паролей

Backend должен хранить пароль пользователя в хешированном виде.

### FR-S-007. Отправка результата партии

Backend должен предоставлять защищенный endpoint для сохранения результата партии.

Endpoint должен принимать:

- survivedSeconds;
- mobsKilled;
- resourcesGathered;
- daysSurvived.

Backend должен связать результат с текущим авторизованным пользователем.

### FR-S-008. Расчет score

Backend должен рассчитывать итоговый score результата партии по формуле:

```text
score = survivedSeconds + mobsKilled * 50 + resourcesGathered * 5 + daysSurvived * 100
```

### FR-S-009. Общий лидерборд

Backend должен предоставлять публичный endpoint для получения топа результатов.

Endpoint должен поддерживать параметр `limit`.

Результаты должны быть отсортированы по score в порядке убывания.

### FR-S-010. Личная статистика игрока

Backend должен предоставлять защищенный endpoint для получения статистики текущего пользователя.

Ответ должен включать:

- лучший результат пользователя;
- позицию пользователя в лидерборде, если она определена;
- последние партии пользователя.

### FR-S-011. Хранение данных

Backend должен хранить данные пользователей и результатов партий в PostgreSQL.

## 6. Backend API

### 6.1. Auth API

| Метод | Endpoint | Доступ | Назначение |
| :--- | :--- | :--- | :--- |
| POST | `/api/auth/register` | Публичный | Регистрация пользователя |
| POST | `/api/auth/signin` | Публичный | Вход пользователя |
| POST | `/api/auth/refresh` | Публичный | Обновление access token |
| GET | `/api/auth/me` | JWT | Получение текущего пользователя |

### 6.2. Leaderboard API

| Метод | Endpoint | Доступ | Назначение |
| :--- | :--- | :--- | :--- |
| POST | `/api/leaderboard/results` | JWT | Сохранение результата партии |
| GET | `/api/leaderboard/top?limit=N` | Публичный | Получение топа результатов |
| GET | `/api/leaderboard/me` | JWT | Получение личной статистики |

## 7. Требования к данным

### 7.1. Пользователь

Пользователь должен содержать:

- id;
- username;
- password hash;
- role.

### 7.2. Результат партии

Результат партии должен содержать:

- id;
- user;
- score;
- survivedSeconds;
- mobsKilled;
- resourcesGathered;
- daysSurvived;
- createdAt.

### 7.3. Игровое состояние клиента

Игровое состояние должно содержать:

- игрока;
- список объектов карты;
- список мобов;
- камеру;
- время суток;
- состояние активности партии;
- состояние завершения партии;
- визуальные эффекты атаки;
- статистику партии.

## 8. Ограничения текущей версии

Текущая MVP-версия описывает только реализованный одиночный игровой режим и backend для авторизации и лидерборда.

Документ не включает требования к функциям, которые отсутствуют в текущей реализации.
